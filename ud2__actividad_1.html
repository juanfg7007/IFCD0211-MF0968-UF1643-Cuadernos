UF1643. UD2. Actividad 1. Caso práctico grupal: Determinar los tipos de sistema de gestión de información y gestores de datos


Enunciado (Descripción)

Esta actividad se realizará de manera grupal, para ello tu tutor-formador creará distintos grupos de trabajo y te indicará a qué grupo perteneces, los miembros que lo forman y el espacio de la plataforma que se habilitará para que podáis desarrollar la actividad.

Preguntas / Actividades a realizar 

Actividad 1:
A. Indica el tipo de sistema de gestión de información, atendiendo a sus objetivos.
B. Indica el tipo de sistema de gestión de información, desde el punto de vista empresarial.


Actividad 2:
C. Indica el tipo de sistema de gestión de información, según el entorno de aplicación.
D. Indica el tipo de sistema de información, según número de usuarios.
E. Indica si el tipo de sistema de información, según el número de sitios.


Actividad 3:
F. Indica el sistema gestor de datos, según el modelo de datos (DBMS).
G. Indica las características de las transacciones del gestor de datos.


Actividad 4:
H. Indica que interfaz de usuario usa el intérprete de comandos del gestor de datos.
I. Indica el tipo de sistema de gestión de base de datos.


Actividad 5:
1. En un barrio de Cádiz, han abierto una nueva academia especializada en estudios universitarios y oposiciones, los alumnos demandan un sitio web donde se puedan registrar y encontrar todo el temario que les van enseñando en sus clases. 
2. Luego del derrame petrolero del buque OlegNaydenov que se incendió y fue remolcado a 15 millas al sur de la Isla Gran Canaria, donde se hundió el 14 de abril del 2015 cargado de 1.400 toneladas de combustible, el gobierno quiere desarrollar un sistema de información donde se mida con un sensor y publique en línea el nivel de contaminación de las playas que pudieran verse afectadas por este y otros incidentes.
3. Una organización no gubernamental quiere desarrollar un sistema de información apoyado en el software ATLAS.ti para cuantificar el incumplimiento de las promesas electorales. ¿Qué tipo de sistema de gestión de información es el más adecuado para cada caso?

Una vez que todos los grupos de trabajo realicen la tarea encomendada, el tutor-formador recopilará toda la información aportada por los distintos grupos de trabajo, creando un único documento donde se recogerán todos los aspectos trabajados en esta actividad, y lo subirá a la plataforma para que todos los alumnos puedan hacer uso del mismo.

UD2 - Actividad 1 - Resolución

Pulsa aquí para visualizar: UD2 - Actividad 1 - Resolución

Amazon

  1. Sistema de gestión de comercio electrónico.
  2. Sistema de información empresarial (ERP) y de gestión de clientes (CRM).
  3. Aplicación web global.
  4. Millones de usuarios a nivel global.
  5. Múltiples sitios internacionales, distribuidos en varios países.
  6. Amazon DynamoDB, Amazon RDS (varios motores de bases de datos como MySQL, PostgreSQL).
  7. Transacciones distribuidas, consistencia eventual y fuerte, escalabilidad horizontal.
  8. Interfaz web, API, CLI (Command Line Interface).
  9. Base de datos NoSQL (DynamoDB), relacional (RDS).

Facebook

  1. Sistema de gestión de redes sociales.
  2. Sistema de gestión de relaciones con el cliente (CRM), análisis de datos y publicidad.
  3. Aplicación web global.
  4. Miles de millones de usuarios activos.
  5. Múltiples centros de datos globales.
  6. MySQL, Apache Cassandra, HBase.
  7. Alta disponibilidad, replicación de datos, consistencia eventual.
  8. API, CLI, interfaz web interna para administración.
  9. Base de datos relacional (MySQL), NoSQL (Cassandra, HBase).

Sistema de Información Schengen de segunda generación (SIS II)

  1. Sistema de información policial y judicial.
  2. Sistema de gestión de seguridad y control fronterizo.
  3. Aplicación gubernamental europea.
  4. Decenas de miles de usuarios, principalmente autoridades policiales y judiciales.
  5. Múltiples sitios en varios países miembros de la UE.
  6. Oracle Database.
  7. Alta seguridad, consistencia transaccional, recuperación ante desastres.
  8. Interfaz web segura, API, CLI para administración.
  9. Base de datos relacional.

Gobierno del Estado de Jalisco

  1. Sistema de información gubernamental.
  2. Sistema de gestión administrativa y de servicios públicos.
  3. Aplicación gubernamental regional.
  4. Miles de usuarios, incluyendo ciudadanos y empleados gubernamentales.
  5. Múltiples oficinas y dependencias del gobierno estatal.
  6. MySQL, SQL Server.
  7. Consistencia transaccional, alta disponibilidad.
  8. Interfaz web, API, CLI para administración.
  9. Base de datos relacional.

Sistema de Información Electoral, Parlamento de Andalucía

  1. Sistema de gestión electoral.
  2. Sistema de gestión de procesos electorales y resultados.
  3. Aplicación gubernamental regional.
  4. Miles de usuarios, incluyendo votantes y autoridades electorales.
  5. Múltiples sitios en la región de Andalucía.
  6. Oracle Database.
  7. Alta seguridad, consistencia transaccional, auditoría de transacciones.
  8. Interfaz web, API, CLI para administración.
  9. Base de datos relacional.

Sistema de Información Geográfica Nacional

  1. Sistema de información geográfica (GIS).
  2. Sistema de gestión de datos geoespaciales.
  3. Aplicación gubernamental nacional.
  4. Miles de usuarios, incluyendo investigadores y autoridades.
  5. Múltiples sitios a nivel nacional.
  6. PostgreSQL con PostGIS.
  7. Manejo de datos espaciales, consistencia transaccional.
  8. Interfaz web, API, CLI.
  9. Base de datos relacional con extensiones geoespaciales.

Servicios Municipales de Barcelona

  1. Sistema de gestión de servicios públicos.
  2. Sistema de gestión administrativa y de servicios municipales.
  3. Aplicación gubernamental municipal.
  4. Miles de usuarios, incluyendo ciudadanos y empleados municipales.
  5. Múltiples oficinas municipales.
  6. SQL Server.
  7. Consistencia transaccional, alta disponibilidad.
  8. Interfaz web, API, CLI.
  9. Base de datos relacional.

Centro de Salud de San Miguel

  1. Sistema de gestión de salud.
  2. Sistema de gestión de información clínica (HIS).
  3. Aplicación de salud regional.
  4. Centenares de usuarios, incluyendo pacientes y personal de salud.
  5. Un único centro de salud.
  6. MySQL.
  7. Alta seguridad, consistencia transaccional, auditoría de accesos.
  8. Interfaz web, API, CLI.
  9. Base de datos relacional.

Universidad Complutense de Madrid

  1. Sistema de gestión académica.
  2. Sistema de gestión de información educativa (LMS).
  3. Aplicación educativa.
  4. Decenas de miles de usuarios, incluyendo estudiantes, profesores y personal administrativo.
  5. Múltiples campus y edificios universitarios.
  6. Oracle Database, MySQL.
  7. Consistencia transaccional, alta disponibilidad.
  8. Interfaz web, API, CLI.
  9. Base de datos relacional.

Iberdrola

  1. Sistema de gestión de energía.
  2. Sistema de gestión de clientes (CRM), gestión de operaciones (ERP).
  3. Aplicación empresarial.
  4. Miles de usuarios, incluyendo empleados y clientes.
  5. Múltiples oficinas y plantas de energía.
  6. Oracle Database, SQL Server.
  7. Consistencia transaccional, alta disponibilidad.
  8. Interfaz web, API, CLI.
  9. Base de datos relacional.

Analizadores de Carbono / Azufre

  1. Sistema de gestión de análisis químicos.
  2. Sistema de gestión de laboratorio (LIMS).
  3. Aplicación de laboratorio.
  4. Decenas de usuarios, incluyendo técnicos de laboratorio y administradores.
  5. Un único laboratorio o múltiples laboratorios dentro de una organización.
  6. MySQL.
  7. Consistencia transaccional, seguridad de datos, manejo de grandes volúmenes de datos analíticos.
  8. Interfaz web, API, CLI.
  9. Base de datos relacional.

Academia en Cádiz

  1. Sistema de Gestión de Contenidos (CMS): su objetivo principal es la gestión y distribución de contenido educativo.
  2. Educativo: orientado a la gestión de recursos educativos y la interacción con estudiantes.
  3. Local/Educativo: aplicado en un entorno educativo específico, inicialmente en un barrio de Cádiz.
  4. Decenas a cientos de usuarios: los alumnos y el personal académico de la academia.
  5. Unisitio: inicialmente para una única academia.
  6. MySQL o PostgreSQL: ambos son adecuados para gestionar bases de datos relacionales con estructuras de datos simples y robustas.
  7. ACID (Atomicidad, Consistencia, Aislamiento, Durabilidad): para asegurar la integridad y fiabilidad de las transacciones de datos.
  8. Web y CLI (Command Line Interface): los administradores pueden usar CLI para tareas avanzadas y los usuarios finales interactúan a través de la interfaz web.
  9. Relacional: adecuado para la gestión estructurada de datos educativos.

Sistema de Información para medición de contaminación en Gran Canaria

  1. Sistema de Información Ambiental (EIS): su objetivo es monitorear y reportar el nivel de contaminación ambiental.
  2. Gobierno/medioambiental: utilizado por entidades gubernamentales para la gestión ambiental.
  3. Regional/nacional: aplicado a nivel regional, en las playas de Gran Canaria.
  4. Miles de usuarios: incluye personal gubernamental, científicos y público en general.
  5. Multisitio: implementado en varias playas y puntos de monitoreo.
  6. PostgreSQL/PostGIS: adecuado para la gestión de datos geoespaciales y ambientales.
  7. ACID para transacciones críticas y eventual consistency para datos de sensores: asegura la integridad en operaciones críticas y flexibilidad en la captura de datos de sensores.
  8. Web y CLI: la interfaz web para visualización de datos y CLI para gestión avanzada.
  9. Relacional con soporte GIS: adecuado para manejar datos espaciales y atributos ambientales.

Sistema de Información para incumplimiento de promesas electorales

  1. Sistema de análisis de textos cualitativos: enfocado en el análisis y cuantificación de textos relacionados con promesas electorales.
  2. Investigación y análisis: utilizado por la ONG para investigación y análisis político.
  3. Internacional: podría aplicarse en diversos contextos políticos y geográficos.
  4. Decenas a cientos de usuarios: investigadores y analistas de la ONG.
  5. Unisitio o multisitio: dependiendo del alcance de la investigación y ubicación de los investigadores.
  6. ATLAS.ti y MySQL/PostgreSQL: ATLAS.ti para análisis cualitativo y una base de datos relacional para el almacenamiento estructurado.
  7. ACID: para asegurar la integridad de los datos analizados y almacenados.
  8. Web y CLI: web para visualización y análisis de datos, y CLI para administración.
  9. Relacional: adecuado para manejar los datos estructurados derivados del análisis cualitativo.
